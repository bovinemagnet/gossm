= Configuration
Paul Snow

:description: gossm configuration reference covering environment variables, config file format, and AWS profile setup.

gossm is configured through a layered system that lets you set sensible defaults in a shared
config file and override them per-project or per-invocation with environment variables.

== Configuration Precedence

Settings are resolved in the following order. Later sources override earlier ones:

[cols="1,3"]
|===
|Priority |Source

|1 (lowest)
|Compiled defaults — values baked into the gossm binary at build time.

|2
|`~/.gossm/config` — personal config file in your gossm home directory.

|3
|`.env` in the current working directory — project-level overrides. gossm reads this file
automatically if it exists in the directory from which you run the command.

|4 (highest)
|Environment variables — set in the shell or injected by a secrets manager.
|===

This means you can define your usual port and log level in `~/.gossm/config`, override them for
a specific project in a `.env` file, and further override them on the command line without
touching either file.

== Environment Variables

[cols="2,3,1"]
|===
|Variable |Description |Default

|`GOSSM_PORT`
|TCP port on which the daemon serves the web dashboard.
|`8877`

|`GOSSM_LOG_LEVEL`
|Verbosity of log output. Accepts `debug`, `info`, or `warn`.
Lower levels include all messages from higher levels (i.e. `debug` includes `info` and `warn`).
|`warn`

|`GOSSM_PID_DIR`
|Directory where gossm writes the PID file (`gossm.pid`) and Unix domain socket (`gossm.sock`).
The directory is created automatically if it does not exist.
|`~/.gossm`
|===

=== Example ~/.gossm/config

[source,shell]
----
GOSSM_PORT=8877
GOSSM_LOG_LEVEL=warn
GOSSM_PID_DIR=/home/myuser/.gossm
----

=== Example .env (project-level)

[source,shell]
----
# Increase logging verbosity for this project
GOSSM_LOG_LEVEL=debug

# Use a different dashboard port to avoid clashing with another service
GOSSM_PORT=9443
----

NOTE: The `.env` file uses the same `KEY=VALUE` format as the `~/.gossm/config` file.
Lines beginning with `#` are treated as comments and ignored.

== AWS Profiles

gossm does not manage AWS credentials itself — it delegates to the AWS CLI and the standard
AWS SDK credential chain. The two files you need to configure are:

`~/.aws/config`::
Defines named profiles. Each profile specifies the AWS region, the IAM role to assume, and
the source profile (the base credential) to use.

`~/.aws/credentials`::
Stores the base AWS access key ID and secret access key for each named base account.

=== ~/.aws/config

[source,ini]
----
[profile SAMPLE-PROFILE] <1>
region = ap-southeast-2 <2>
role_arn = arn:aws:iam::1234567890:role/FullAWSAccess <3>
source_profile = sample-admin-account <4>
----
<1> The profile name used by gossm (passed to `-p` or set as `AWS_PROFILE`).
<2> The AWS region for API calls made under this profile.
<3> The ARN of the IAM role to assume when using this profile.
    This role must have `ssm:StartSession` and `ec2:DescribeInstances` permissions.
<4> The name of the entry in `~/.aws/credentials` that provides the initial credentials used to
    assume the role above.

=== ~/.aws/credentials

[source,ini]
----
[sample-admin-account] <1>
aws_access_key_id = ABCD... <2>
aws_secret_access_key = ABCD... <3>
----
<1> Matches the `source_profile` value in `~/.aws/config`. This name can also be used directly
    with `gossm connect -p sample-admin-account` if no role assumption is required.
<2> The AWS access key ID. Obtain this from the AWS IAM console or your organisation's
    identity provider.
<3> The AWS secret access key corresponding to the access key ID above.

IMPORTANT: Never commit `~/.aws/credentials` to version control or share it with others.
Treat the secret access key with the same care as a password.

=== Multiple Profiles

You can define as many profiles as needed. gossm will display them all in the interactive
profile picker when `AWS_PROFILE` is not set.

[source,ini]
----
[profile production]
region = ap-southeast-2
role_arn = arn:aws:iam::111111111111:role/SREAccess
source_profile = sso-base

[profile staging]
region = ap-southeast-2
role_arn = arn:aws:iam::222222222222:role/DeveloperAccess
source_profile = sso-base

[profile sandbox]
region = us-east-1
role_arn = arn:aws:iam::333333333333:role/AdminAccess
source_profile = personal-account
----

=== AWS SSO

If your organisation uses AWS IAM Identity Centre (SSO), your `~/.aws/config` profiles will look
slightly different:

[source,ini]
----
[profile production-sso]
sso_start_url = https://my-org.awsapps.com/start
sso_region = ap-southeast-2
sso_account_id = 111111111111
sso_role_name = SREAccess
region = ap-southeast-2
----

You must authenticate the SSO session before using the profile:

[source,bash]
----
aws sso login --profile production-sso
----

TIP: See the https://docs.aws.amazon.com/cli/latest/userguide/sso-configure-profile-token.html[AWS documentation]
for full details on configuring SSO-based profiles.

== Required IAM Permissions

The IAM role or user associated with the profile needs at minimum:

[source,json]
----
{
  "Effect": "Allow",
  "Action": [
    "ssm:StartSession",
    "ssm:TerminateSession",
    "ssm:DescribeSessions",
    "ssm:GetConnectionStatus",
    "ec2:DescribeInstances"
  ],
  "Resource": "*"
}
----

Restrict the `Resource` field to specific instance IDs or SSM document ARNs for tighter security
in production environments.
